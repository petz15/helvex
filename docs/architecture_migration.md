# Zefix Analyzer — Architecture Migration Plan

## Current State

FastAPI monolith + Jinja2 server-rendered UI + PostgreSQL + in-process background job thread + ML pipeline (TF-IDF/HDBSCAN + spaCy) + Claude Haiku AI classification. Deployed via Docker Compose with Nginx reverse proxy.

---

## Target Architecture: Modular Microservices on Kubernetes

### Core Principles
- Split along natural seams in the existing codebase
- Each service is independently deployable
- Async communication (message queue) for long-running jobs
- Sync REST for latency-sensitive queries (UI data fetching)
- Shared PostgreSQL with separate schemas (pragmatic over pure microservices dogma at this scale)

---

## Services

### 1. `frontend` — Next.js
- Replaces Jinja2 templates
- Next.js App Router: SSR where needed, client-side interactivity elsewhere
- Components: dashboard, company detail, job queue, settings, Leaflet map
- Communicates with `api` service via REST/fetch
- Can be deployed to Vercel (free tier) or co-hosted in K8s

### 2. `api` — FastAPI (API Gateway / BFF)
- Thin layer: JWT authentication, routing, rate limiting
- Exposes full OpenAPI spec (auto-generated by FastAPI)
- Delegates work to workers by publishing to Redis Streams
- Sources from: `app/ui/routes.py`, `app/crud/`, `app/models/`

### 3. `collection-worker` — Data Ingestion Service
- Consumes jobs from Redis Streams queue
- Handles: Zefix bulk import, Google Search batches, geocoding, company detail fetches
- Sources from: `app/services/collection.py`, `app/api/zefix_client.py`, `app/api/google_search_client.py`, `app/api/geocoding_client.py`
- Stateless: can scale horizontally for parallel canton imports

### 4. `scoring-worker` — Scoring & Enrichment Service
- Consumes jobs from Redis Streams queue
- Handles: Zefix score recalculation, website match scoring, Claude Haiku classification
- Sources from: `app/services/scoring.py`, Claude batch logic in `app/services/collection.py`
- Isolated so scoring logic can evolve independently from data collection

### 5. `ml-worker` — ML Pipeline Service
- Consumes jobs from Redis Streams queue
- Handles: TF-IDF vectorization, HDBSCAN/K-Means clustering, spaCy NLP preprocessing
- Sources from: `app/services/cluster_pipeline.py`
- Heavy dependencies (scikit-learn, spaCy, pandas) isolated in its own container image
- Triggered on-demand from UI or on schedule

## Infrastructure

### Cloud Provider: Hetzner Cloud
- European/Swiss data residency for GDPR compliance with Swiss customers
- ~3–5× cheaper than AWS/GCP for equivalent compute
- Full setup cost: ~€34/month vs ~€150+ on AWS
- Strong Kubernetes support via Hetzner Cloud Controller Manager (CCM) and CSI driver

**Alternative:** AWS EKS if maximum global employer recognition is the priority.

### Orchestration: K3s (Lightweight Kubernetes)
- Production-grade Kubernetes without the operational overhead of full K8s
- One Helm chart per service
- HPA (Horizontal Pod Autoscaler) on collection-worker and scoring-worker for bulk import load spikes

### Message Queue: Redis Streams
- Replaces the existing in-process thread worker (`app/ui/routes.py` job worker)
- Redis also serves as cache and rate-limit store
- Managed option: Upstash Redis (serverless, pay-per-use, works with Hetzner)

### Database: PostgreSQL
- Keep existing schema and all 16 Alembic migrations
- Option A: Hetzner Managed Databases (less ops overhead)
- Option B: CloudNativePG operator in K8s (more control)
- Shared database, separate schemas per service (pragmatic choice at this scale)

### Object Storage: Hetzner Object Storage (S3-compatible)
- Geocoding SQLite (~300–400 MB)
- CSV exports
- spaCy model artifacts
- TF-IDF vectorizer pickles

### CDN / Edge: Cloudflare (free tier)
- TLS termination, DDoS protection, asset caching
- Replaces the current Nginx TLS role at the edge

## Migration Phases

### Phase 1 — Extract Frontend (lowest risk, highest visual impact)
1. Create `services/frontend/` as a new Next.js project
2. Add REST API routes to the existing FastAPI for all data the UI needs
3. Run Next.js in parallel with existing Jinja2 UI — no breakage
4. Migrate routes one by one: dashboard → detail → jobs → settings → map

### Phase 2 — Introduce Message Queue
1. Add Redis to `docker-compose.yml`
2. Create a thin queue producer in `api` service (publish job specs to Redis Streams)
3. Extract workers as separate Python processes that consume from the queue
4. Replace the existing in-process thread worker in `app/ui/routes.py`

### Phase 3 — Separate Container Images
1. Create `services/collection/Dockerfile`, `services/scoring/Dockerfile`, `services/ml/Dockerfile`
2. Each shares the same alembic migrations and database connection config
3. Update `docker-compose.yml` to launch all services locally

### Phase 4 — Deploy to K3s on Hetzner
1. Provision K3s cluster via Hetzner Cloud
2. Write Helm chart per service
3. Set up GitHub Actions: build → push to GHCR → helm upgrade

### Phase 5 — Multi-tenancy (when customer onboarding begins)
1. Add `tenant_id` column to `companies`, `users`, `job_runs` tables
2. Enable Row-Level Security (RLS) in PostgreSQL
3. Issue tenant-scoped JWT tokens with tenant claim

## Open-Source Considerations

### What signals quality to employers and contributors
- **OpenAPI docs** at `/docs` (FastAPI generates automatically) — expose publicly
- **GitHub Actions CI** — lint (ruff, eslint), test, build Docker images on every push
- **Helm charts** — demonstrates K8s deployment knowledge
- **Architecture Decision Records** in `docs/adr/` — shows architectural thinking
- **Docker Compose** for local dev — single command to run the full stack
- **Semantic versioning + CHANGELOG.md** — professional OSS hygiene

### Secrets handling for open-source
- `.env.example` with all required variables documented (no real values)
- API keys (Zefix, Serper, Anthropic) injected via K8s Secrets or Doppler/Vault
- Geocoding SQLite downloaded at container startup from object storage (not committed to git)

## Files to Modify During Migration

| File | Migration Action |
|---|---|
| `app/main.py` | Becomes `services/api/main.py` — remove Jinja2/UI routes |
| `app/ui/routes.py` | Split: API routes stay in `api`, job dispatch moves to queue publisher, UI routes removed |
| `app/services/collection.py` | Move to `services/collection/` — refactor as queue consumer |
| `app/services/scoring.py` | Move to `services/scoring/` — refactor as queue consumer |
| `app/services/cluster_pipeline.py` | Move to `services/ml/` — refactor as queue consumer |
| `docker-compose.yml` | Add Redis service, add worker services, keep working throughout |
| `Dockerfile` | Replace single Dockerfile with per-service Dockerfiles |
| `nginx/` | Remove — Cloudflare handles edge TLS; K8s Ingress handles internal routing |
| `alembic/` | Shared across services — lives in `services/api/` or a shared `migrations/` package |
